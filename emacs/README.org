Personal frustration and the biggest time waster in my life (comparable to video
games). Please turn away and retain your sanity. Otherwise you might end up
tracking all your tasks, notes, and thoughts in Emacs.

This README is the high-level tour: what this configuration is, why it looks the
way it does, and how the pieces fit together. If you just want to run it, skip to
[[*Getting it running][Getting it running]].

* A short history

I did not start here. For years I ran [[https://github.com/syl20bnr/spacemacs][Spacemacs]] and helped maintain it. It is a
lovely project, but over time I wanted something smaller, something I could hold
in my head completely and change without fighting a framework.

Then there is [[https://github.com/doomemacs/doomemacs][Doom Emacs]]. Doom is great and very opinionated, which is not a bad
thing on its own. But it is heavily vim-driven, and it carries a whole layer of
machinery (and its own mental model) that I did not want to keep paying for. I do
not edit modally, so most of that would be complexity I carry without ever using.

So this is the third option: roll my own. Small, legible, mine. The goal was
never to be clever. The goal was a setup I fully understand, where nothing
happens that I cannot point to.

* The idea

The whole thing is built around one decision: treat the configuration as a real
Emacs package, not a pile of scripts.

Once you commit to that, a lot follows almost for free:

- *Fast to load.* Startup stays under a second. Packages load lazily and garbage
  collection is tuned during startup (see [[file:early-init.el][early-init.el]] and [[file:init.el][init.el]]).
- *Clean byte-compilation.* Everything compiles with warnings treated as errors,
  so a typo or a bad call is a build failure now, not a surprise at runtime three
  weeks from now.
- *Linted.* checkdoc and friends keep the source honest.
- *Tested.* Real libraries get real tests.
- *Autoloaded.* Personal functions are available everywhere without me wiring up
  ~require~ by hand.
- *Extensible.* Adding a feature or a library is a small, local change.

The rest of this document is really just the consequences of that one decision.

* How it fits together

** Three kinds of files

Everything lives in =lisp/=, and every file is one of three kinds, marked by its
prefix:

- =init-*.el= feature configuration. This is where packages get configured with
  =use-package=, lazily. [[file:init.el][init.el]] requires each of these in order. Examples:
  [[file:lisp/init-editor.el][init-editor.el]], [[file:lisp/init-ui.el][init-ui.el]], [[file:lisp/init-vulpea.el][init-vulpea.el]].
- =config-*.el= plain constants and variables. Cheap to load, safe to require
  from anywhere, no package loading or heavy work. Example: [[file:lisp/config-path.el][config-path.el]].
- =lib-*.el= personal libraries. These are *not* required directly. Public
  functions are marked =;;;###autoload=, an autoloads file is generated at build
  time, and they become available everywhere on demand. Examples: [[file:lisp/lib-string.el][lib-string.el]],
  [[file:lisp/lib-buffer.el][lib-buffer.el]], [[file:lisp/lib-vulpea.el][lib-vulpea.el]].

A library is just a normal file with autoload cookies:

#+begin_src emacs-lisp
;;; lib-greeting.el --- Tiny example -*- lexical-binding: t; -*-

(require 'dash) ; lib files may require their dependencies freely

;;;###autoload
(defun greeting-shout (&rest words)
  "Upcase WORDS and join them with spaces."
  (string-join (-map #'upcase words) " "))

(provide 'lib-greeting)
;;; lib-greeting.el ends here
#+end_src

After adding it, =./setup.sh install= regenerates the autoloads and
=greeting-shout= is available everywhere, no =require= needed.

** Eldev: the config is a package

[[https://github.com/doublep/eldev/][Eldev]] is what makes "treat the config as a package" real. The [[file:Eldev][Eldev]] file declares
=init.el= as the package main file and pulls every =lisp/*.el= into the project.
From there Eldev gives me:

- byte-compilation with warnings as errors,
- linting,
- the [[https://github.com/jorgenschaefer/emacs-buttercup][buttercup]] test suite,
- generation of the autoloads file from the =lib-*.el= sources,
- a clean, isolated environment where dependencies live separately from my actual
  Emacs.

Day to day I do not call Eldev directly: [[file:setup.sh][setup.sh]] wraps it (and a [[file:Makefile][Makefile]] wraps
that). The same commands run in CI, so the badge up top is just "does it still
compile, lint, and test".

** elpaca: packages

Packages are installed with [[https://github.com/progfolio/elpaca][elpaca]], bootstrapped in [[file:lisp/init-elpa.el][init-elpa.el]]. It is, in my
opinion, the best package manager Emacs has today: fast, asynchronous, installs
from ELPA, MELPA, and arbitrary git repos, and integrates cleanly with
=use-package=.

Guix and Nix can also manage Emacs packages, but I do not go that route here and
this README does not get into it. (If you are curious where my head is at on Nix,
I wrote about it [[https://www.d12frosted.io/posts/2025-11-25-nix][here]].)

* Notes, tasks, everything: vulpea

My whole org-mode life runs on [[https://github.com/d12frosted/vulpea][vulpea]], a tool I built for exactly this. It keeps
its own database and *is not org-roam*; please do not reach for org-roam commands
or assumptions when reading this part.

The org side is a small stack of vulpea packages plus a few helpers:

- *vulpea* the core: notes, links, the database, capture, the moving parts of a
  personal knowledge base.
- *[[https://github.com/d12frosted/vulpea-ui][vulpea-ui]]* the visual layer: sidebar, backlinks with previews.
- *[[https://github.com/d12frosted/vulpea-journal][vulpea-journal]]* dated notes and journaling.
- *[[https://github.com/d12frosted/vulpea-para][vulpea-para]]* opinionated PARA-style defaults: the agenda dispatcher, project
  and task predicates, capture templates.
- *consult-vulpea* notes through consult.

On top of that sit the usual org niceties (capture, refile, attach, agenda,
download, toc) and a couple of my own touches for margins and emphasis markers.
It all lives in [[file:lisp/init-vulpea.el][init-vulpea.el]].

One detail worth calling out: with more than ten thousand org files, scanning all
of them to build the agenda is far too slow. Instead the agenda file list is
derived from the database (the files that actually carry open work) and refreshed
as I go. I have written about this approach over a series of posts, starting
[[https://d12frosted.io/posts/2021-01-16-task-management-with-roam-vol5.html][here]].

* UI: stable on purpose

I do not chase Emacs eye-candy. I want a UI that looks the same next year as it
does today, with just enough air that it feels modern and is easy on the eyes.
Concretely ([[file:lisp/init-ui.el][init-ui.el]] and [[file:early-init.el][early-init.el]]):

- *Theme:* [[https://github.com/protesilaos/modus-themes][modus]] by Protesilaos, with a few palette and heading tweaks. =<f5>=
  toggles light and dark.
- *Fonts:* Iosevka via [[https://github.com/protesilaos/fontaine][fontaine]], with presets for different sizes.
- *Breathing room:* [[https://github.com/protesilaos/spacious-padding][spacious-padding]], wide window dividers, rounded undecorated
  frames, internal borders. This is the "modern" part, and it is all spacing, not
  chrome.
- *Quiet mode line:* [[https://github.com/tarsius/minions][minions]] folds away the minor-mode noise.
- *Small comforts:* pulsar briefly pulses the line I jump to, indent-bars draws
  subtle indentation guides, hl-line, no blinking cursor, no bell.

That is the whole story. No dashboards, no animations, nothing that moves unless
I asked it to.

* Keys

Keybindings go through [[https://github.com/noctuid/general.el][general.el]] with =M-m= as the leader prefix
([[file:lisp/init-kbd.el][init-kbd.el]]). It is the Spacemacs leader idea without the modal editing: plain
Emacs keys, no evil, no vim states to track. =which-key= shows what lives under a
prefix as I type.

* Private bits

Some configuration is private (machine-specific or secret) and does not belong in
this repo. If =EMACS_PRIVATE_DIR= is set, Eldev links any =*.el= files from there
into =lisp/= before building, so private files take part in the same compile and
autoload flow as everything else without ever being committed.

* Getting it running

** Requirements

- Emacs 30.2 or newer
- Eldev (installed automatically by =setup.sh= if missing)

** Install

From the dotfiles root:

#+begin_src bash
./eru.sh install emacs
#+end_src

Or straight from this directory:

#+begin_src bash
cd emacs
./setup.sh install
#+end_src

** Commands

[[file:setup.sh][setup.sh]] is the single entry point, and =make= targets mirror it:

| Command | What it does                               |
|---------+--------------------------------------------|
| install | Bootstrap packages and generate autoloads  |
| upgrade | Upgrade all packages                       |
| compile | Byte-compile everything (warnings = errors) |
| lint    | Run checkdoc and regexp linting            |
| test    | Run the buttercup suite                    |
| clean   | Remove generated files                     |
| doctor  | Health check the setup                     |

Commands chain, so my usual pre-commit pass is:

#+begin_src bash
./setup.sh compile lint test
#+end_src

** A note on linting

=package-lint= is intentionally off. It exists to police ELPA packages (prefix
naming, file headers, and so on), none of which applies to a personal
configuration. checkdoc and byte-compilation do the real work here. If you ever
see package-lint-style complaints about prefixes, something is misconfigured.

In-editor, flycheck uses the standard =emacs-lisp= checker with =lisp/= and the
elpaca packages added to its load path, so =init-*= files can find each other and
=lib-*= files can find their dependencies without false errors.
